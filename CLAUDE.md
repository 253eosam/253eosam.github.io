# 개인 학습 노트 시스템

이 프로젝트는 Claude AI와 대화하며 공부한 내용을 **책의 글처럼 읽기 쉬운 형태**로 정리하는 개인 학습 노트 공간이다. 나중에 다시 읽었을 때 빠르게 이해할 수 있는 고품질 노트를 목표로 한다.

## 목적

- 공부한 내용을 빠짐없이 상세하게, "책의 한 챕터"처럼 정리
- 다시 읽었을 때 자연스러운 흐름으로 빠르게 이해할 수 있는 구조
- Claude AI와의 대화를 통한 개념 정리 및 이해도 심화
- 학습 내용의 체계적 관리 및 아카이브

## 폴더 구조

```
/
├── CLAUDE.md                 # 프로젝트 가이드라인 (이 문서)
├── content/
│   ├── YYYY-MM-DD-토픽명/    # 각 노트 폴더
│   │   └── index.md          # 노트 본문
│   └── ...
```

## 워크플로우

### 새 노트 작성

1. `content/YYYY-MM-DD-토픽명/` 폴더 생성
2. 폴더 안에 `index.md` 파일 생성 (아래 메타데이터 포함)
3. Claude AI와 대화하며 내용 정리 및 보완
4. commit/push하면 자동 배포

```markdown
---
title: '노트 제목'
date: 'YYYY-MM-DD'
category: '카테고리명'
tags: ['태그1', '태그2']
description: '한줄 요약'
status: 'completed'
---
```

## Claude AI의 역할

- 사용자의 거친 메모, 질문, 키워드를 **책의 한 챕터**처럼 자연스럽고 체계적인 글로 변환
- 읽는 흐름이 자연스럽도록 문단 간 연결을 유지
- 필요한 곳에 다이어그램, 표, 코드 예시를 자동으로 삽입
- 누락된 개념이나 보충 설명이 필요한 부분을 능동적으로 제안

## 노트 구조 원칙

모든 학습 노트는 다음 구조를 기본으로 한다. 책의 한 챕터를 읽듯 자연스럽게 읽히도록 작성한다.

### 필수 구성 요소

1. **제목 + 한줄 정의**
   - 이 주제가 무엇인지 1-2문장으로 명확하게 정의

2. **목차 (TOC)**
   - 글 상단에 배치하여 전체 구조를 한눈에 파악
   - 섹션 링크를 통해 원하는 부분으로 바로 이동 가능

3. **개요**
   - 이 개념을 왜 알아야 하는지, 어떤 문제를 해결하는지 설명
   - 독자가 "읽을 가치가 있다"고 느낄 수 있도록 동기 부여

4. **핵심 개념 (본문)**
   - 정의 → 원리 → 상세 설명 순서로 논리적 흐름 유지
   - 공부한 모든 내용을 빠짐없이 기록
   - 비교 대상이 있으면 표로 정리하여 차이점을 명확히
   - 문단 간 자연스러운 연결로 "읽는 흐름" 유지

5. **실전 예시**
   - 코드 예시는 복사해서 바로 실행 가능한 완전한 코드로 작성
   - 코드 전후에 무엇을 하는 코드인지, 결과가 어떤지 설명

6. **시각 자료**
   - 복잡한 흐름이나 구조는 Mermaid 다이어그램으로 시각화
   - 텍스트만으로 이해하기 어려운 부분은 반드시 그림으로 보완
   - 예시:
     ```mermaid
     graph LR
       A[입력] --> B[처리] --> C[출력]
     ```

7. **Q&A (인라인 배치)**
   - 공부하면서 떠오른 질문과 답변을 본문의 문맥에 맞는 위치에 배치
   - 형식: `> **Q: 질문 내용**` 블록인용 사용
   - 예시:
     ```markdown
     ## 클로저란?
     클로저는 함수와 그 함수가 선언된 렉시컬 환경의 조합이다...

     > **Q: 클로저는 메모리 누수를 일으키나?**
     > 클로저가 참조하는 변수가 GC되지 않으므로 주의가 필요하다.
     > 불필요한 참조는 null로 해제하는 것이 좋다.
     ```

8. **요약 (Summary)**
   - 글 하단에 핵심 내용을 bullet point로 간결하게 정리
   - 요약만 읽어도 전체 내용이 떠올릴 수 있도록 작성

## 작성 스타일

### 1. 책의 글 스타일

- **자연스러운 서술**: 단순 나열이 아닌, 앞뒤 맥락이 연결되는 서술형 문장
- **정의 우선**: 각 개념은 명확한 정의로 시작
- **계층적 구조**: 큰 개념 → 세부 개념 → 예시 순서로 깊어지는 구조
- **크로스 레퍼런스**: 관련 개념이 다른 노트에 있으면 참조 링크 추가
- **독립적 섹션**: 각 섹션을 독립적으로 읽어도 이해할 수 있게 작성

### 2. 상세함의 기준

- **생략 금지**: "자세한 내용은 공식 문서 참고" 같은 생략 표현 사용하지 않음
- **모든 단계 기록**: 설정, 설치, 실행 과정을 한 단계도 빠짐없이 기록
- **용어 설명**: 처음 등장하는 전문 용어는 반드시 간단한 설명 추가
- **코드 전체**: 코드 스니펫이 아닌, 복사해서 바로 실행 가능한 완전한 코드

### 3. 가독성 원칙

- **짧은 문단**: 한 문단은 3-4문장 이내로 유지
- **표 활용**: 비교, 옵션, 설정값 등은 표로 정리
- **코드 블록**: 모든 코드에 언어 태그 명시 (```js, ```python 등)
- **굵은 글씨**: 핵심 키워드나 중요 개념은 **굵은 글씨**로 강조
- **들여쓰기**: 하위 개념은 들여쓰기로 계층 표현
- **여백 활용**: 섹션 사이, 코드 블록 전후, 표 전후에 빈 줄을 넣어 시각적 구분을 확보한다. 빽빽하게 이어진 텍스트보다 적절한 여백이 있는 글이 읽기 쉽다.
- **문장 단위 줄바꿈**: 긴 문단은 의미 단위로 줄을 나눈다. 한 문단 안에서도 독립된 내용이면 줄바꿈으로 분리하여 한눈에 들어오게 한다.

### 4. 톤 가이드라인

- **교과서/기술서적 톤**: 사실 중심, 감정적 표현이나 주관적 평가 최소화
- **간결한 문체**: "~이다", "~한다" 체로 통일
- **과도한 이모지 사용 금지**
- **마무리 멘트 금지**: "도움이 되었으면", "공유해주세요" 등 불필요한 마무리 멘트 추가하지 않음
- **불필요한 소감 섹션 금지**: "개인적인 생각", "마지막 한 마디" 등 추상적인 소감/전망 섹션 추가하지 않음
- **Humanizer 스킬 적용**: 글 작성 시 `/humanizer` 스킬을 참고하여 AI가 생성한 느낌을 제거한다. 과장된 상징 표현, 홍보성 언어, 모호한 귀속, em dash 남용, 3의 법칙 남발, AI 특유의 어휘, 불필요한 접속 부사 등을 피한다.

### 5. 구조 품질

- **깨진 표 수정**: 마크다운 표 형식 정확히 작성
- **명확한 섹션**: 내용에 맞는 적절한 제목 구조 (h2 → h3 → h4 순서 준수)
- **읽기 쉬운 형식**: 표, 코드 블록, 인용문, 다이어그램 적절히 활용
- **논리적 흐름**: 읽는 사람 입장에서 자연스러운 정보 순서

- 오타 및 문법 검토 완료
- frontmatter 메타데이터 완성
- 목차와 실제 섹션 구조 일치
- 모든 전문 용어에 설명 포함
